The scenario

Zero-Day Discovery: How it works

Five independent network nodes train local IDS models on IID data — every client holds all 15 attack classes, including SQL injection. At round 5, Client 0 receives an 80 % surge of fresh SQL injection traffic and the federation must absorb it. We compare FedAvg (uniform averaging) vs FedDiv (divergence-weighted aggregation).

Evaluation methodology

Server-Side Held-Out Evaluation

Before any training begins, 10 % of the dataset (≈57,290 samples, stratified across all 15 classes) is locked as a server-side test set. No client ever accesses these samples. After each FL round, the aggregated global model is evaluated here — every metric shown in the timeline below comes from truly unseen data.

Held-out server eval — 10 %

≈57,290 stratified samples. Never exposed to any client. After each FL round, the aggregated global model is tested here to generate the F1 scores in the timeline.

Client training data — 88.1 %

≈504,699 samples split IID across 5 clients (≈100,940 each). All 15 attack classes present on every node from round 1.

Discovery pool (SQL injection) — 1.9 %

10,912 SQL injection samples withheld from Client 0. Released at round 5 to simulate a real zero-day surge, triggering gradient divergence and propagation.

Design rationale

FedAvg vs FedDiv — Design Choices

Both algorithms solve the same federated learning problem: train a shared intrusion-detection model across heterogeneous edge nodes without centralising raw traffic data. Their design philosophies diverge at a single critical question — should the server treat all client updates as equally trustworthy?

FedAvg — Uniform Aggregation

Core assumption: all clients observe the same data distribution. Aggregation weights are fixed proportionally to local dataset size — the server plays a purely passive role.

  • Aggregation: weighted mean of parameter tensors, weights ∝ |Di|
  • Server role: stateless — stores no knowledge between rounds
  • Client sampling: balanced mini-batches maintained throughout
  • Heterogeneity: not modelled — clients are treated as i.i.d. workers
  • New-class handling: relies on the averaging process to propagate new knowledge; minority signal is divided by the number of clients
  • Privacy model: no server-side data required at any point
FedDiv — Probe-Weighted Aggregation

Core assumption: after a zero-day event, clients diverge in expertise. The server holds a small, labelled probe set and uses it to measure each client's competence on the target class before aggregating.

  • Aggregation: wi ∝ divip, where divi = mean softmaxSQL(fi(xprobe))
  • Server role: active — maintains a probe set and EMA divergence history
  • Client sampling: switches to natural (unbalanced) sampling post-discovery to expose true data distribution
  • Heterogeneity: explicitly exploited — a specialist client receives proportionally higher weight
  • New-class handling: probe score surfaces the specialist; power p controls how sharply weight concentrates
  • Privacy model: server holds ≤2k labelled samples (alarm-triggered, not raw telemetry); client data never leaves the node
